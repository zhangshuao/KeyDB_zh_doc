# Gopher 支持
 
 KeyDB包含一个Gopher协议的实现，如RFC 1436中所指定的。

Gopher协议在90年代后期非常流行，它是web的替代品，服务器端和客户端的实现都非常简单，KeyDB服务器只有100行代码来实现这种支持。

你现在怎么对付Gopher？好吧，Gopher从来没有真正死亡，最近有一个运动，为了使Gopher更层次的内容，仅仅由纯文本文件组成的复活。
一些人想要一个更简单的互联网，另一些人则认为主流互联网变得过于受控制，为想要新鲜空气的人创造一个替代空间是很酷的。

## 它是怎么工作的？

KeyDB Gopher支持使用KeyDB的内联协议，特别是两种无论如何都是非法的内联请求：空请求或以"/"开头的任何请求（没有以斜杠开头的KeyDB命令）。
正常的RESP2/RESP3请求完全脱离了Gopher协议实现的路径，通常也会被服务。

如果在启用Gopher时打开到KeyDB的连接并向其发送类似"/foo"的字符串，则如果存在名为"/foo"的key，则通过Gopher协议提供该key。

## 安全警告

如果您计划将KeyDB放在服务器Gopher页面的公共访问地址中，请确保为实例设置密码。设置密码后：

    1.Gopher服务器（如果启用，默认情况下不会）将通过Gopher杀死服务内容。
    2.但是，在客户端进行身份验证之前，不能调用其他命令。

所以使用requirepass选项来保护您的实例。

要启用Gopher支持，请使用以下配置行。

    gopher-enabled yes

访问不是字符串或退出的key将在Gopher协议格式中产生错误。
